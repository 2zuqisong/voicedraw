# voice-draw 设计文档

> 版本 0.1.0 | 2026-06-14

---

## 1. 项目目标

通过中文语音控制画布绘图。用户说话，AI（LLM）理解意图并调用工具生成图形。

核心原则：
- 自然语言输入，零学习成本
- LLM tool-calling 驱动，而非模板匹配
- 矢量 + 像素双模式，各司其职
- 本地优先（Tauri 桌面应用），同时支持浏览器访问

### 2.2 计划但未实现

#### 2.2.1 语音相关

| 计划 | 原因 |
|------|------|
| 流式语音识别（边说边出字） | Web Speech API `continuous: true` 已启用，但 `interimResults` 在中文下不够流畅，需要换用专用 ASR 服务 |
| 离线语音识别 | Web Speech API 依赖浏览器云端服务，离线需要集成 Whisper.cpp 或类似本地模型 |
| 多语言支持 | 当前 system prompt 和语音识别均为中文，英文需要独立 prompt |
| 语音唤醒词 | 需要持续监听，增加功耗和隐私顾虑，当前点击触发已满足需求 |

#### 2.2.2 画布能力

| 计划 | 原因 |
|------|------|
| 自由绘画（鼠标手绘） | 与"语音控制"核心定位冲突，像素模式已完全移除手动绘制 |
| 图层管理 | 当前节点可以上下移动但无显式图层概念，需求场景不明确 |
| 动画/过渡效果 | Fabric.js 支持但 LLM 工具调用模型下动画序列定义复杂 |
| 协作绘图 | 需要 WebSocket + 多人状态同步，属于 v2 功能 |
| 无限画布缩放 | Fabric.js 自带缩放但缺少 UI 控件（缩放滑块、小地图等） |
| 节点旋转 | Fabric.js 支持但 LLM 工具未定义旋转参数 |

#### 2.2.3 LLM 能力

| 计划 | 原因 |
|------|------|
| 流式响应（token by token） | async-openai 支持 `chat_stream` 但 scheduler 多轮工具调用架构需要较大重构 |
| 本地小模型（Ollama 等） | async-openai 支持 OpenAI 兼容 API，Ollama 可用但精度不足以支撑复杂工具调用 |
| 多模态（图片输入） | 需要视觉模型理解画布截图，异步上下文和成本较高 |
| 对话记忆跨会话 | 当前对话历史仅存内存，刷新后丢失，需要持久化方案 |

#### 2.2.4 像素模式扩展

| 计划 | 原因 |
|------|------|
| 像素动画（逐帧） | 帧序列数据结构 + 播放器需要独立设计，与当前架构差异大 |
| 更多表情（20+） | 8 个表情已覆盖常用场景，扩展需要手动设计像素图案 |
| 像素喷涂/抖动 | 单色填充已满足基本需求，复杂像素效果工具定义成本高 |
| 像素导出为 ICO/GIF | 使用场景有限，PNG 导出已满足基本需求 |

#### 2.2.5 用户体验

| 计划 | 原因 |
|------|------|
| 操作历史面板 | 当前只有 undo/redo 按钮，无可视化历史列表 |
| 画布缩略图导航 | 大画布时缺乏全局视图，需要 minimap |
| 深色/浅色主题自动切换 | 系统级跟随，当前手动切换 set_theme |
| 快捷键 | 桌面端常用操作（Ctrl+Z 撤销等）未绑定 |
| 拖拽移动节点 | Fabric.js 原生支持拖拽，但拖拽后的位置不会同步回 Rust CanvasState |

## 3. 架构决策记录

### 3.1 为什么用 Tauri 而不是纯 Web

- Tauri 提供系统级 IPC（invoke），前端直接调用 Rust 函数
- 字体渲染（Noto Sans SC）在系统 WebView 中更稳定
- 文件导出（PNG/SVG）使用系统对话框
- 同时支持浏览器模式（axum HTTP API）用于开发调试

### 3.2 为什么用 Fabric.js 而不是原生 Canvas/SVG

- 对象模型（Group、Textbox、Polyline 等）天然匹配节点/连线抽象
- 内置事件系统（选中、拖拽、缩放）
- 导出 SVG 直接调用 `toSVG()`
- 社区成熟，v6 支持 TypeScript

### 3.3 为什么用 Zustand 而不是 Redux

- 轻量（<1KB），零样板代码
- 支持在 React 组件外读写（`useAppStore.getState()`），适合 Canvas 初始化场景
- TypeScript 类型推导友好

### 3.4 为什么 LLM tool-calling 而不是规则匹配

- 自然语言变化无穷，规则匹配覆盖率永远不够
- "画一个红色圆形，右边放一个蓝色矩形" 涉及 2 个节点 + 相对定位 + 颜色 + 批量创建，单一规则无法处理
- LLM 可以处理歧义、反问确认、增量修改
- 代价：2-5 秒延迟（已通过复杂度预判和进度推送优化）

### 3.5 为什么双模式（矢量/像素）要物理隔离

- system prompt 混合会导致 LLM 选错工具（像素模式下用矢量工具，反之亦然）
- 对话上下文混合会导致 LLM 提及另一模式的内容
- 解决方案：
  - system prompt 按模式切割（矢量 LLM 看不到像素章节，反之亦然）
  - 画布摘要按模式过滤（矢量模式不显示像素数据）
  - 对话上下文在切换模式时清除

### 3.6 为什么像素表情用程序化生成而非位图

- 16×16 位图存储需要 256 个格子 × 8 个表情 = 大量硬编码数据
- 程序化生成（圆形 + 轮廓 + 特征点）代码量小且易维护
- 后续可扩展到不同尺寸（12×12、24×24）无需重新设计

## 4. 数据流

```
用户语音 → Web Speech API → 文本
    │
    ▼
预处理（Rust preprocessor）
    ├── 快捷指令（撤销/重做/清空/导出）→ 直接执行
    └── 需要 LLM → LLM Scheduler
                    │
                    ├── 复杂度预判（像素/简单指令跳过）
                    ├── System prompt（按模式选择）
                    ├── LLM API 调用（DeepSeek/OpenAI）
                    ├── 工具调用执行（pixel_rect / add_node / ...）
                    ├── CanvasState 更新
                    └── 进度推送（每轮工具调用后 emit canvas-progress）
    │
    ▼
前端渲染（CanvasView / PixelCanvas）
```
